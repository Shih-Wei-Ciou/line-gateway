---

## 1. 專案願景

打造一套**以 LINE 為主要互動介面**的宮廟數位平台。對信眾而言，無需下載 App，加入 LINE 官方帳號即可訂閱關注的廟宇、收到活動通知、線上回應參加意願。對廟方而言，能夠系統化發送活動資訊、收集報名、管理出席，取代過去依賴 LINE 群組的混亂作業。

---

## 2. 系統定位與分工

整體系統由**兩個子系統**組成：

### 2.1 廟方管理系統（TEP — 已存在）

由團隊另一位成員開發完成，職責包括：

- 廟宇基本資料管理
- 活動建立、編輯、發布
- 報名名單管理
- 廟方管理員後台介面
- 提供 RESTful API 讓外部系統對接

技術棧：Flask + MySQL + React。

### 2.2 LINE 互動層（本期建置範圍）

**本人負責的部分**，職責包括：

- LINE 官方帳號訊息收發
- 信眾與廟宇的訂閱關係管理
- 活動通知推播（Flex Message）
- 信眾在 LINE 內的互動回應處理
- 將互動結果同步至 TEP 系統

技術棧：Python + Flask + PostgreSQL，部署於 Zeabur 雲端平台。

### 2.3 為什麼分兩個子系統

廟方管理是**內部後台場景**，重點在功能完整、操作流暢；LINE 互動是**對外接觸點**，重點在訊息設計、推播時機、回應流暢度。兩者技術需求不同、迭代節奏不同。透過 HTTP API 串接，各自獨立演進。

未來若新增其他互動入口（如獨立 App、Web），只需新增對應的對接層，不影響既有系統。

## 3. 三方角色

| 角色 | 介面 | 場景 |
|---|---|---|
| 信眾 | LINE App | 訂閱廟宇、收通知、回應報名、查詢自己的紀錄 |
| 廟方管理員 | TEP 網頁後台 | 建活動、發推播、看名單、管理廟宇資料 |
| 系統營運者 | LINE 對接層後台 | 監控推播狀況、查詢使用者、處理異常 |

## 4. 核心使用情境

### 情境 A：信眾首次加入

信眾掃描廟宇提供的 QR Code 加入 LINE 官方帳號 → 系統自動建立使用者紀錄 → 推送歡迎訊息引導至訂閱頁 → 信眾選擇關注哪些廟宇。

### 情境 B：廟方發布活動

廟方管理員在 TEP 後台建立活動（標題、時間、地點、名額） → 點選「發送 LINE 通知」 → TEP 通知 LINE 對接層 → 對接層查詢訂閱該廟的信眾 → 批次推送 Flex Message 卡片。

### 情境 C：信眾回應報名

信眾在 LINE 收到活動卡片 → 點擊「我要參加」按鈕 → 系統首次會請信眾填寫姓名電話（之後不再要求） → 報名資料寫入 TEP 系統 → 信眾收到確認訊息 → 廟方在 TEP 後台立即看到報名清單更新。

### 情境 D：信眾取消報名

信眾透過 LINE 選單打開「我的報名」頁面 → 看到所有已報名的活動 → 點擊取消 → 系統呼叫 TEP 取消 API → 推送確認訊息。

## 5. 關鍵設計決策

### 5.1 多對多訂閱關係

從第一天起，系統設計就支援「一位信眾訂閱多間廟」與「一間廟有多位訂閱者」。即使初期只有一間示範廟，這個結構讓未來擴充新廟時，使用者體驗無縫銜接（不用再加另一個 LINE 帳號）。

### 5.2 單一 LINE 官方帳號

整個平台只用一個 LINE 官方帳號。信眾加一次好友即可訂閱多間廟，避免「每間廟都要加一個 LINE」的繁瑣。

### 5.3 LINE 對接層全面接管 LINE 通訊

過去 TEP 內建的 LINE 廣播功能將由 LINE 對接層完全取代。所有從系統發出去的 LINE 訊息、收到的 LINE 事件，統一由對接層處理。這保證了訊息一致性，也讓 LINE 互動體驗能集中優化。

### 5.4 資料來源權責清晰

- **廟宇、活動、報名等業務資料**：唯一存放於 TEP
- **LINE 使用者資訊、訂閱關係**：唯一存放於 LINE 對接層

兩邊不複製對方的資料。需要時透過 API 即時查詢。這避免了資料同步衝突的常見痛點。

### 5.5 不重建已有功能

LINE 對接層**不重建**廟方後台、活動管理、報名管理等 TEP 已實作的功能。這個原則嚴守，讓本期專案能在 4 週內完成。

       └──────────────────────────┘

### 6.2 兩個關鍵 API 對接點

**A：TEP → LINE 對接層**（推播觸發）
廟方在 TEP 後台按下「發送通知」時，TEP 呼叫對接層的內部 API，由對接層執行 LINE 推播。

**B：LINE 對接層 → TEP**（資料寫入）
信眾在 LINE 完成報名或取消時，對接層呼叫 TEP 的公開 API 寫入資料。

### 6.3 部署架構

| 環境 | 平台 | 用途 |
|---|---|---|
| 開發 | 本機（Docker + ngrok） | 快速迭代、功能測試 |
| 正式 | Zeabur 雲端平台 | 真實使用者連線 |

開發採「本地優先」原則：所有功能在本機完整測試通過後，才推送到雲端正式環境。雲端與本地各自獨立的資料庫，避免測試資料汙染。

## 7. 資料模型（LINE 對接層）

四張核心表，刻意保持輕量：

| 資料表 | 用途 |
|---|---|
| `line_users` | LINE 使用者基本資訊與聯絡資料 |
| `subscriptions` | 多對多訂閱關係 |
| `action_logs` | 操作流水紀錄（除錯與稽核用） |
| `push_logs` | 推播紀錄與成效追蹤 |

不存放廟宇、活動、報名等資料 — 這些屬於 TEP 的權責。

## 9. 開發方法論

### 9.1 分階段測試

整個 MVP 切分為 **10 個小階段**，每階段都是可獨立 demo 的小功能。完成一階段、測試通過、修完 bug，才進入下一階段。避免「全部做完再測試」的傳統陷阱。

### 9.2 本地優先

每個功能先在本機完整測試。本機不過的功能不上雲端。本機環境包含：開發中的對接層、Docker 跑的 TEP（與正式版本一致）、本地資料庫、ngrok 提供的對外通道。

### 9.3 漸進式整合

LINE 對接層與 TEP 之間的 API 對接，按使用情境逐一打通。先讓 LINE 內部的互動跑起來（看得到、按得到），再串接 TEP 寫入資料，最後處理取消與查詢。

## 10. 風險與對策

| 風險 | 對策 |
|---|---|
| 設計稿尚未產出 | MVP 先用 LINE 官方預設樣式 + 簡潔配色，功能可獨立推進 |
| 兩個子系統需協調 API | 建立清楚的對接清單，每週對齊一次 |
| 時程壓力 | 嚴格遵守分階段交付，必要時砍次要功能 |
| LINE 推播免費額度限制 | 每月 500 則對單廟測試足夠；正式上線多廟時再評估付費方案 |
| 未來新增廟宇 | 多對多訂閱架構從第一天設計，不需重構 |

## 11. 預期成果

4 週後可呈現的具體成果：

1. **可運作的 LINE 官方帳號**：信眾能加入、能訂閱、能收到活動通知、能在 LINE 內報名與取消
2. **完整的雙向資料流**：廟方在 TEP 後台的操作能透過 LINE 觸達信眾；信眾在 LINE 的回應能即時反映在 TEP 後台
3. **清晰的系統文件**：架構書、API 對接文件、操作說明
4. **可擴充的基礎**：多對多訂閱、Adapter 預留外部整合介面
