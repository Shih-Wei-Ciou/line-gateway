00這份文件解決什麼

問題：專案在 2026-05-26 經歷了一次重大架構轉向（從「LINE 為主」改成「Web App 為主」）。但轉向後，md/ 資料夾裡所有舊文件描述的還是舊架構，而且白紙黑字寫著「不做 Next.js / 不做 Docker / 不做信眾 Web App」——跟現在實際在做的事完全矛盾。

轉向後的「現況」只散落在三個地方：HTML task log（記每次做了什麼）、轉向盤點 log（記新舊差異）、Web App 那邊的架構書（只講 Web App）。沒有一份文件完整講清楚三個系統（Web App + LINE + 宮廟後台 TEP）的全貌、為什麼這樣選、整個專案在做什麼。

這份文件就是補上那個缺口。它是「現況總綱」——開新對話、新人加入、或自己三個月後回來，看這一份就懂全貌。所有數字與名稱都已對照三個 repo 的實際 code 校正過。

重要 · 舊文件作廢通知
md/新對話視窗_Onboarding包.md、md/宮廟數位互動平台_整體架構_給顧問版.md、md/首要目標.md 描述的都是舊架構（LINE 為主、拒絕 Web App）。它們是歷史紀錄、不是現況。以本文件為準。

01專案在做什麼

緣引拜拜通是一個宮廟數位轉型專案。讓信眾能線上探索廟宇、查神明、看參拜指南、依需求找「求○○該拜誰」、收藏關注廟宇、（規劃中）報名活動，並透過 LINE 收到關注廟宇的活動通知。

這是一個體驗型專案，不是功能型專案：

功能型（不是這個）
像報稅系統——使用者完成任務就走、過程越短越好。
體驗型（這個）
使用者產生情感連結、想再回來。每個功能要讓人覺得儀式感、被理解感、內心平靜。
核心信念 · 警惕遊戲化陷阱
不做積分、徽章、稀有度這類遊戲化機制——不適合廟宇場景，會違反核心信念。

內容現況（依 Web App seed 校正）

內容數量說明
神明（deities）10 位三官大帝、媽祖、文昌、財神、土地公、保生大帝、註生娘娘、玄天上帝、城隍、月老（seed 已建，規劃擴充）
需求標籤（needs）12 類姻緣 / 財富 / 學業 / 事業 / 健康 / 求子 / 平安 / 除煞 / 太歲 / 安神 / 家庭 / 其他
廟宇（temples）12 間5 間示範（已發布）+ 7 間 core 骨架（draft、待田調）
拜拜指南（guides）規劃中seed 尚未含指南內容

示範主力廟：白河三官寶殿。另有霧峰南天宮等。

02架構轉向（舊 → 新）

2026-05-26 的關鍵決定：從「LINE 為主」轉向「Web App 為主」。這是理解整個專案的關鍵。

舊架構（已淘汰）
LINE 是主要介面。信眾在 LINE 內用 LIFF 完成所有事：訂閱、報名、查詢。

明確拒絕：React / Next.js / Docker / 信眾 Web App。

LINE 對接層（Flask）直接打 TEP。
新架構（現況）
Web App 是主場。信眾真正在 Web App 完成所有事。

LINE 變成「招牌 + 通知通道」——降低門檻、發推播、把人導去 Web App。

擁抱 Next.js + Docker + Web App。
一句話比喻：LINE 像招牌、Web App 是店、TEP 是倉庫
LINE 像星巴克的官方帳號——你不會在 LINE 裡喝咖啡，但每次它推新品、卡片做得讓你想點進去。
Web App 是「店」本身——所有真正能逛、能看、能做的事都在這。
TEP 是「進貨倉庫」——活動、報名這些資料未來的家（學長另一個既有的廟方後端平台）。

為什麼轉向

舊架構的問題新架構怎麼解
LINE 內 LIFF 能做的體驗有限Web App 可以做完整的視覺、互動、地圖、參拜引導
所有功能擠在 LINE 裡、難以擴充Web App 是完整系統、要加功能很自由
難以做 SEO、被搜尋到Web App 有獨立網址、可被 Google 收錄
非 LINE 用戶完全無法使用Web App 任何人都能開、Google 登入也行

三句話定位

系統一句話對信眾
Web App（緣引拜拜通）信眾真正在用的主場唯一會看到完整內容的地方

關鍵釐清 · 三件容易搞混的事
① Web App 目前自帶廟/神/指南內容（存在自己的 yuanyin MySQL），不是即時從 TEP 抓。跟 TEP 的整合是在「報名」階段才發生（打 TEP 的 public 報名 API）。

② TEP 自己也內建 LINE（有 line_users 表、/api/line/webhook、一個 LIFF app）。新架構下，LINE 的統一窗口是 Max 的 LINE Gateway，取代散在 TEP 裡的 LINE 功能，避免兩套 LINE 打架。

③ 「Web App」≠「TEP」。它們是兩個不同的 repo、不同的 MySQL、不同前端框架（Next.js vs React19）。Web App 是新的信眾體驗層，TEP 是既有的廟方營運平台。

04為什麼這樣選（關鍵決策）

用 LINE 當招牌、不重做 App
信眾年齡層偏大、LINE 覆蓋率高。LINE 負責「敲門通知」，真正服務在 Web App。
Web App 用 Next.js 14（App Router）
學長已經會、已經做了不少。SSR + RSC 對 SEO 好、體驗像 App。
地圖用 Google Maps（@vis.gl/react-google-maps）
需要 Places 評論數→知名度、和台灣在地圖資準確度。（早期試過 Leaflet，已改用 Google Maps；Leaflet 套件殘留但未使用。）
對接 TEP、不重建廟方後端
學長已有 TEP（活動/報名/廟方營運）。避免重工，報名階段透過 public API 對接。
Web App 資料庫用 MySQL（不是 PostgreSQL）
學長的 templeuser 與 TEP 本來都用 MySQL 8。統一生態、避免兩套。
用 Docker 跑開發環境
學長已把 MySQL + Flask + Next.js 包成 docker-compose 一鍵啟動。（注意：舊架構曾拒絕 Docker，新架構推翻）
用 Zeabur 部署
台灣節點、$5/月、GitHub 一鍵自動部署。
帳號用 LINE Login + Google Login
LINE Login 為主（綁了才能收推播）、Google 為輔（門檻低、跨裝置）。email 重複用「提示綁定」策略（C 方案）。
Web App 後端用 Flask 模組化自動發現
加新功能 = 加一個模組資料夾（manifest + routes + models），自動掛載，不動別人的 code。
LINE Gateway 獨立 repo
LINE 對接的職責跟 Web App 不同。獨立的 line-gateway repo，透過 API 跟 Web App 互通。

雙層認證（兩套完全隔離的 JWT）

後台 admin會員 member
守門@require_admin（auth 模組）@require_member（members 模組）
token 放哪JWT in httpOnly cookieJWT Bearer / localStorage lyz_member_token
claimsrole=super_admin, temple_scope=*kind=member, memberId, lineUserId

資安紅線 · ADMIN_AUTH_DISABLED
ADMIN_AUTH_DISABLED=1 是本機測試後門（後台免登入）。正式上線（Zeabur）絕對不可設，不設＝預設要登入。

07帳號系統（已完成 · Max 核心成果）

Web App 原本只有 demo 假登入。Max 在 feat/line-auth branch 加上完整的 LINE Login + Google Login OAuth。

登入流程

信眾 │ │ 點「使用 LINE 登入」或「使用 Google 登入」（白底品牌按鈕） ▼ Flask 後端 /api/members/oauth/{line|google}/authorize │ 種 oauth_state cookie（防 CSRF）→ 導向 OAuth 授權頁 ▼ LINE / Google 伺服器 → 使用者按「允許」→ 回 callback 帶 code+state │ ▼ Flask 後端 /callback │ ① 驗 state（比對 cookie） │ ② 用 code 換 id_token │ ③ 驗 id_token → 拿 UID/sub + 名字 + email（Google 還驗 audience） │ ④ 帳號合併檢查（email 衝突 → C 方案提示綁定） │ ⑤ upsert member → 簽 member JWT │ ⑥ 導回前端 /?token=eyJ... ▼ Next.js 前端 │ MemberProvider 讀 URL token → 存 localStorage(lyz_member_token) │ 打 /api/members/me → 驗 JWT → 顯示名字 ▼ 信眾看到「○○○ · LINE 會員」🎉
關鍵設計 · 登入按鈕必須用「絕對網址」
其他 API 走相對 /api（Next 同源代理避 CORS）。但 OAuth 例外：authorize 種的 oauth_state cookie 必須由 callback 同一個 origin 讀回驗證，所以登入按鈕直接指後端 origin，不能走代理（否則永遠 invalid_state）。

帳號合併 C 方案

同一個 email 先用 Google 註冊、後來想用 LINE 登入（或反過來）——系統不自動合併（避免誤判），而是提示「此 email 已用 X 註冊，要綁定嗎？」使用者同意才把第二個 OAuth 識別寫進同一個帳號。對應 bind 端點。

關鍵端點（前綴 /api/members）

MethodPath用途
GET/oauth/line/authorize導去 LINE 授權
GET/oauth/line/callbackLINE 回來後處理 + 簽 JWT
POST/oauth/line/bind補綁 LINE（C 方案）
GET/oauth/google/authorize導去 Google 授權
GET/oauth/google/callbackGoogle 回來後處理 + 簽 JWT
POST/oauth/google/bind補綁 Google（C 方案）
GET/me用 member JWT 取得自己資訊
dev 設定 · BACKEND_INTERNAL_URL 必設
本機 frontend/.env.local 必須設 BACKEND_INTERNAL_URL=http://localhost:5001，否則 Next 代理會 fallback 到正式站 Zeabur 後端 → 本地簽的 token 被不同 JWT secret 驗 → 401。

08宮廟後台 TEP 詳解

TEP（Temple Events Platform，資料庫 amulet_system_v2）是學長既有的一套完整廟方營運平台，比 Web App 更早存在、規模更大。新架構把它定位成「倉庫」——活動 / 報名 / 廟方營運資料的源頭，Web App 在報名階段透過它的 public API 對接。

它是什麼 · 服務誰

角色介面場景
廟方管理員廟方管理後台（frontend）/ 官網 CMS（sgbd）建活動、發推播、看名單、管理廟宇/商品/訂單/營收
信眾信眾端 App（user-web，目前 Mock）/ LIFF打卡集點、兌換、活動報名
系統管理員super admin平台級監控

四個前端 + 一個後端

子專案是什麼狀態
backendFlask REST API（~105 endpoints，Swagger /api/docs/）~95%
frontend廟方管理後台（儀表板、圖表）成熟
sgbd廟宇官網管理 CMS（公告、活動、點燈、進香）完整
user-web信眾端 AppMock 未串接後端
liffLINE LIFF（活動報名）單廟 VITE_TEMPLE_ID=1

TEP 自己的認證（注意：跟 Web App 是另一套）

JWT 雙 token（1 小時 access + 7 天 refresh，refresh_tokens 表可撤銷）。三種帳號表 + 三個守門裝飾器：

裝飾器誰能過
@token_required任何登入者
@temple_admin_token_required廟方管理員
@super_admin_token_required系統管理員

整合合約 · Web App / LINE Gateway 怎麼接 TEP

關鍵是 TEP 開了一組免登入 public 端點，直接吃 lineUserId，不需要先綁定帳號——這就是 Stage 4 當初的重大發現（報名不卡在 UID 綁定）。

MethodPath用途

釐清 · TEP 內建 LINE vs Max 的 LINE Gateway
TEP 自帶 line_users 表、/api/line/webhook、一個 LIFF app。新架構下由 Max 的 LINE Gateway 當統一 LINE 窗口（招牌 + 推播），TEP 內建的 LINE 功能逐步讓位，避免兩套 LINE webhook / 兩個官方帳號打架。報名資料的「家」仍是 TEP（透過上面 public API 寫入）。

10資料模型（三系統）

① Web App（MySQL yuanyin · templeuser）

members（會員 · Max 擴充過） ├── id, display_name(預設「訪客」), avatar_url ├── line_user_id (UK, nullable, idx) ← LINE 登入 ├── google_sub (UK, nullable, idx) ← Google 登入（Max 加） ├── email (UK, nullable, idx) ← 帳號合併用（Max 加） ├── provider (NOT NULL, 預設 demo) ← {line, google, demo} ├── push_enabled (NOT NULL, 預設 true) ← 推播開關（Max 加） └── created_at, last_seen_at 註：DB 無 CHECK 約束；app 邏輯保證 line_user_id 一定有值（訪客用 demo_ 前綴） wishes（許願 · 學長做） member_id(FK), temple..., status, remind_at favorites（收藏廟宇 · 學長做） member_id(FK), temple 複合唯一 temples / deities / guides / needs（內容 · 學長做） analytics_events（行為追蹤 · IP 經 SHA256+salt 雜湊不存原始 IP）

② LINE Gateway（PostgreSQL · line-gateway）

line_users line_uid(UK), display_name, follow_status ... action_logs 操作流水（稽核 / debug，JSONB payload） push_logs 推播紀錄（flex_payload, status, sent_at） subscriptions：已決定改用 Web App 的 favorites，不在這裡存

③ TEP（MySQL amulet_system_v2 · 30+ 表，擷取關鍵）

public_users / temple_admin_users / super_admin_users（三種帳號） temples 廟宇（座標、打卡半徑、main_deity ...） temple_events 活動（title, start/end, capacity, status） event_registrations 報名（line_user_id nullable idx, user_id nullable） line_users TEP 自己的 LINE 使用者表 amulets / checkins / energy_logs 打卡集點 products / redemptions / addresses 兌換電商 lamp_types / lamp_applications 點燈 notifications / temple_notifications ... 通知（5 表）
注意 · 訂閱/收藏資料的歸屬
新架構下「關注廟宇」用 Web App 的 favorites 表。LINE Gateway 推播時，向 Web App 查「廟 X 有哪些關注者的 LINE UID」（規劃中的內部 API），自己不存訂閱關係。
