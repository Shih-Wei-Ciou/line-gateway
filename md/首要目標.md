一、 我們現在優先要完成的目標是什麼？
目前的最高指導原則是：「先用 LINE 系統解決廟方最痛的行政與金流問題，快速收集數據並建立信任，後續再引導至專屬 App」
。
首要完成的三大核心系統目標為：
整合報名系統：取代目前混亂的 LINE 群組報名，讓信眾透過 LINE 直接選擇時段報名志工或法會活動，後台自動彙整名單
。
數位香油錢與點燈（金流串接）：迎合現代人無現金習慣，開發線上點燈與隨喜捐獻功能，並支援 LINE Pay 等行動支付
。
廟方數據與資訊管理後台：為第一間合作的示範廟宇（如霧峰南天宮）建立一個後台，讓他們能看到報名統計、財務數據，以及上架最新的活動資訊
。
二、 這些目標是怎麼規劃出來的？（策略與邏輯）
這些目標的規劃是基於「使用者問卷」與「實地廟宇訪談」的痛點交叉比對而來：
基於 B2B（廟方端）痛點：霧峰南天宮在訪談中明確指出，目前志工招募與排班會面臨人力不足的困擾，且他們於近期（1/20）剛談好正在申請 LINE Pay
。因此，系統必須優先幫他們解決「人（報名排班）」與「錢（線上香油）」的管理問題。
基於 B2C（信眾端）痛點：問卷調查顯示，年輕人不去廟宇的主因是「不知道怎麼拜、怕做錯」以及「在意現場動線與便利性」
。
降低使用門檻：會議中確認，要求長輩或初次參拜的年輕人立刻下載一個全新的 App 門檻太高。因此規劃先透過大家都有的 LINE 來綁定數位身分，快速體驗服務
。
三、 身為製作資訊系統的人，您該如何開始設計？
作為系統與程式整合負責人，您現在需要停止構思過於複雜的 3D 動畫或龐大的架構，改以「模組化」與「快速對接」為核心。請依照以下步驟開始設計與開發：
Step 1: 基礎環境與 LINE API 建設
申請與設定：先為示範廟宇申請一個專屬的 LINE 官方帳號（OA），並開通 LINE Developer 權限
。
串接 LINE Login 與 LIFF：您的前端介面將不再是原生的 iOS/Android App，而是嵌在 LINE 裡面的微網頁（LIFF）。信眾點擊選單後，系統需能自動抓取 LINE UID 建立會員資料，達到「免密碼登入」的效果
。
Step 2: 建立輕量化後端與資料庫 (MySQL + Flask)
您的資料庫設計需要對應前述的核心目標，優先建立以下關聯表：
Users：以 LINE UID 作為唯一識別，記錄信眾基本資料、累積的福報值
。
Events/Volunteers：儲存廟方發布的法會與志工需求時段、報名人數上限與已報名名單。
Donations：記錄線上香油錢與點燈的交易紀錄，用於串接金流 API（如 LINE Pay 或第三方支付）並回傳成功狀態給廟方後台
。
Step 3: 與設計團隊（UI/UX）界定技術底線
近期會議中顯示，設計端對於前端能呈現的效果（如 3D 導覽、複雜的轉場動畫）存在技術落差與誤解
。您必須主動出擊：
給出規格限制：明確告訴設計負責人，在 LINE LIFF 網頁的框架下，哪些動畫做得到、哪些做不到。請他們先提供靜態圖檔（PNG/SVG）或簡單的 GIF
。
強調長輩友善介面：南天宮特別要求「開發友善模式，字大一點、顏色介面調整」
。您在開發前端網頁時，需與設計師確認版面是否符合高對比與大字體的規範。
