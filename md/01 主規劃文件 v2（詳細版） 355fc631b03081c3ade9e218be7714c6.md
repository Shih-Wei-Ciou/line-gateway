# 一、專案核心定位

## 1.1 一句話定位

本系統是 **LINE 端的對接層**，把 LINE 使用者的互動轉接到既有的 TEP 廟宇後端。本系統**不重建廟宇管理功能**，TEP 仍是廟方管理後台與真實資料的擁有者。

## 1.2 系統本質：服務型而非功能型

這是**體驗型專案**不是**功能型專案**。

|  | 功能型 | 體驗型（本案） |
| --- | --- | --- |
| 例子 | 報稅系統、訂便當 | 寶可夢、星巴克、本案 |
| 目標 | 完成任務後離開 | 產生情感連結後想再回來 |
| 設計 | 過程越短越好 | 過程要有設計 |

核心信念：每一個功能不只是「能用」，而是要讓使用者**覺得這個服務值得花時間停留**。

## 1.3 警惕「遊戲化陷阱」

寶可夢能讓人停留是因為**收集進度與稀有度心理**，星巴克能讓人停留是**累積點數心理**——這些**不適合廟宇場景**。

廟宇的「情感連結」應該是：

- **儀式感**（我被引導走過完整流程）
- **被理解感**（這個服務懂我的需求）
- **內心平靜**（我覺得心情變好了）

不是：

- 我又解鎖了什麼
- 我又集到幾點
- 我又升級了

設計每個功能時，問自己：

- 「這個設計讓使用者感受到的是儀式感，還是只是任務完成的成就感？」
- 「這個機制是讓使用者覺得被理解，還是被誘導？」

如果答案偏向後者，就要重新設計。

# 二、為什麼這樣設計（關鍵決策歷史）

這節記錄專案至今的關鍵架構決策，**為什麼選 A 不選 B**。未來有人問「為什麼這樣做」就指這節。

## 2.1 為什麼選 LINE 而不是獨立 App

**現實考量**：

- 信眾年齡層偏大，不會為了拜拜下載新 App
- 廟方資源有限，無法投資多平台
- LINE 在台灣覆蓋率超高，門檻最低

**LINE 的限制可接受**：

- 介面雖然受限（Rich Menu、LIFF、Flex Message），但夠用
- 推播額度有上限，但單廟測試不會超過
- 功能擴展性比 App 差，但 V1 不需要那麼多功能

## 2.2 為什麼選擇對接 TEP 而不是自建後端

**TEP 已經有的東西**：廟方後台、活動管理、報名系統、廟宇資料庫——這些都已經開發完成。

**本系統重建這些的代價**：

- 多花 2-3 個月開發
- 跟學長系統重複，增加維護成本
- 廟方要學兩套系統

**結論**：本系統定位為**對接層**，TEP 是真實資料來源，本系統只負責 LINE 端獨有的功能（訂閱、推播路由、互動處理）。

## 2.3 為什麼選 PostgreSQL 而不是 MySQL

**TEP 用 MySQL**，但本系統選 PostgreSQL，原因：

- **本系統與 TEP 不直接共用資料庫**，靠 HTTP API 溝通，DB 種類沒影響
- **PostgreSQL 在現代雲端 PaaS 支援更好**（Zeabur、Render、Supabase 都首推）
- **PostgreSQL 處理 JSON 欄位更好**（action_logs、push_logs 都有 JSON 欄位）
- **PostgreSQL 免費額度通常更大**

## 2.4 為什麼選 Zeabur 而不是 GCP/AWS

**Zeabur 的優勢**：

- 台灣節點，使用者延遲低
- 中文介面與支援
- $5/月入門方案，學生友善
- 從 GitHub 一鍵部署

**GCP/AWS 的劣勢**（對本案）：

- 計費複雜，容易超支
- 設定門檻高，浪費時間
- 對小流量專案沒優勢

**長期擴張**：未來若真有商業擴展需求再考慮 GCP/AWS，現階段 Zeabur 完全夠用。

## 2.5 為什麼用 Tailwind + Alpine.js 而不是 React/Vue

**單人開發 + MVP 階段**：

- React/Vue 學習與配置成本高
- npm/webpack 增加部署複雜度
- LIFF 頁面互動不複雜，輕量框架夠用

**Tailwind + Alpine.js**：

- CDN 引入，不用 build 流程
- 學習曲線低
- 視覺還原能力足夠
- 未來真的需要 React 再切換，業務邏輯與 API 都不用動

## 2.6 為什麼用 ngrok 在本地開發

LINE Webhook 必須是 HTTPS，無法用 `http://localhost`。**ngrok 把本地 Flask 變成公開 HTTPS URL**，是本地開發的標準做法。

**雲端開發的代替方案不可行**：

- 直接在雲端開發，每次改一行要等 2-5 分鐘部署，效率慘
- 雲端會被真實使用者打到，bug 會被撞見

結論：**本地優先（ngrok）→ 雲端正式（Zeabur）**。

## 2.7 為什麼 LIFF 建在 LINE Login Channel 而不是 Messaging API

**LINE 在 2023 年改了規定**：LIFF 必須建在 LINE Login Channel 之下，不能建在 Messaging API Channel。

**對本系統的影響**：

- 兩個 Channel 各管各的：Messaging API 管訊息收發、LINE Login 管 LIFF
- 各自有獨立的 Channel Secret
- 環境變數要分開命名

## 2.8 為什麼選擇「先做後端、後對齊設計」

**設計同學是合作夥伴而非外包**，意思是設計討論需要互動，不能單方面 brief。

**等設計討論前的時間怎麼用**：

- 不寫完整 UI（會被改掉重做）
- 寫**技術骨架**（不受設計影響的部分：API、資料庫、LIFF SDK 整合）
- 準備**設計討論材料**（讓討論有起點）

這就是本週「70% D 類 + 30% A 類」的策略由來。

# 三、系統架構

## 3.1 三方角色

| 角色 | 屬於誰 | 介面 |
| --- | --- | --- |
| 信眾 | 使用者 | LINE App + LIFF 頁面 |
| 廟方管理員 | 學長系統 | TEP 管理後台 |
| 中央平台 | 本系統 | LINE Channel + 中央資料庫 |

## 3.2 資料流

**雙向資料流的核心**：

方向 A：**廟方 → 信眾**

```
廟方在 TEP 後台建活動
   ↓
廟方按「LINE 推播」按鈕
   ↓
TEP 呼叫本系統 /internal/push/event
   ↓
本系統查訂閱關係 → 推 Flex Message 給訂閱者
```

方向 B：**信眾 → 廟方**

```
信眾在 LINE 按「我要參加」按鈕
   ↓
本系統收到 postback event
   ↓
本系統呼叫 TEP /api/public/events/X/register
   ↓
TEP 寫入 event_registrations
   ↓
廟方在 TEP 後台看到報名
```

## 3.3 為什麼資料只留在「該負責的系統」

**TEP 擁有的資料**：廟宇、活動、報名（不在本系統重建）

**本系統擁有的資料**：LINE 使用者、訂閱關係、操作流水、推播紀錄

**避免重複的好處**：

- 沒有同步衝突問題
- 廟方在 TEP 改的東西立即生效
- 維護成本低

**代價**：每次顯示廟宇資料時要打 TEP API（稍微慢，但可接受）。

## 3.4 三個入口（本系統提供）

| 入口 | 對象 | 路徑 |
| --- | --- | --- |
| Webhook | LINE Platform | `/webhook/line` |
| LIFF 頁面 | 信眾 | `/liff/*` |
| 內部 API | TEP | `/internal/*` |

注意**沒有「廟方後台」入口**，因為那是 TEP 的範圍。

## Stage 1：LINE Bot 基礎連線 ✅ 已完成

### 在做什麼

讓你的 Flask 能**收到 LINE 的訊息事件**、能**自動回覆**訊息。

### 為什麼需要這個 Stage

- LINE Bot 的所有互動都建立在「webhook 能收到事件」這個基礎上
- 簽章驗證確保只有 LINE 真的打來的請求才會被處理（避免被偽造）
- Echo Bot 是最簡單的「收到→回應」流程，驗證整條路徑能通

### 完成的東西

- ngrok 跑起來，把本機 Flask 變成公開 HTTPS
- LINE Webhook URL 設到 LINE Developer Console
- `/webhook` 端點接收 POST 請求
- X-Line-Signature 簽章驗證（防偽造）
- Echo Bot：傳「hello」會回「你說：hello」
- follow event 處理（加好友時 print 訊息）

### 卡關過的問題

- 簽章驗證一開始失敗：原因是 Channel Secret 填錯（拿了 Channel ID）
- ngrok URL 每次重啟會變：要記得回 LINE Console 改 webhook URL

### 怎麼測試

- 自己加 OA → console 看到「有人 follow」
- 對 OA 傳訊息 → 收到 echo 回應
- curl 假請求 → 被簽章驗證擋掉

## Stage 2：使用者識別與資料庫 ✅ 已完成

### 在做什麼

LINE 使用者進來時，**自動寫進 line_users 表**。每個跟 OA 互動過的人都有檔案。

### 為什麼需要這個 Stage

- 後面所有功能（訂閱、報名、推播）都要先知道「這個使用者是誰」
- LINE 只給你 line_uid，需要透過 Profile API 拿到 displayName 和 pictureUrl
- 處理「使用者封鎖再加好友」這種重複事件，不能重複建 row

### 完成的東西

- follow event → 寫入 line_users（line_uid、display_name、picture_url、follow_status='active'）
- unfollow event → 更新 follow_status='unfollowed'，**不刪除 row**
- 重複 follow 處理（upsert 邏輯：有就更新、沒有就建立）
- LINE Profile API 串接（拿頭像、名字）
- message event → 更新 last_seen_at

### 關鍵設計決策

- **不刪除 row**：保留所有歷史使用者，方便統計「多少人封鎖過」
- **last_seen_at**：未來能看「最近活躍的使用者有誰」
- **Profile API 失敗不 crash**：拿不到頭像就用 'Unknown'，繼續寫入

### 怎麼測試

- 加好友 → DB 有自己的 row、display_name 是 LINE 名稱
- 封鎖再加 → 同一 row、status 從 unfollowed 變回 active
- 傳訊息 → last_seen_at 更新成現在時間

## Stage 3：LIFF + 訂閱機制 🟡 進行中

### 在做什麼

讓使用者透過 **LIFF 頁面（嵌在 LINE 內的網頁）訂閱廟宇**。訂閱後未來活動會推播給他。

### 為什麼需要這個 Stage

- 推播路由的核心是「訂閱關係」：廟方發活動，要推給「訂閱該廟的人」
- 沒有訂閱機制，推播只能「全部都推」或「全部都不推」
- LIFF 是 LINE 內網頁的標準做法，學會這個之後其他 LIFF 功能（報名、我的紀錄）都類似

### 任務分組

**Group A：LINE Console 設定（不寫程式）**

- A1：在 LINE Login Channel 下建立 LIFF 應用
- A2：拿到 LIFF ID、設定 Endpoint URL
- A3：把 LIFF ID 寫進 .env

**Group B：LIFF 前端（最小骨架版）**

- B1：Flask 路由 `/liff/test`
- B2：HTML 頁面 + LIFF SDK 整合
- B3：取得使用者 LINE UID 並顯示
- B4：Fake Mode（電腦瀏覽器打開時用假資料）vs Real Mode（LINE 內用真實資料）
- **不做**：完整 UI、卡片、訂閱按鈕（等設計討論完再做）

**Group C：訂閱 API（純後端）**

- C1：`POST /api/liff/subscriptions` 寫入訂閱
- C2：`DELETE /api/liff/subscriptions/:id` 取消訂閱
- C3：`GET /api/liff/me` 查詢自己的訂閱狀態
- 用 Postman/curl 測試，**不接前端**

### 為什麼分這三組

- A 不會寫程式，純設定，先做避免後面卡住
- B 純前端骨架，學 LIFF SDK 整合，UI 樣式等設計討論完再做
- C 純後端，不受設計影響，做完未來前端來了直接接

### 為什麼這次不做完整 UI

下週（5/8）跟設計同學討論詳細使用流程後才做 UI，避免我猜的版本被丟掉重做。**先做的是技術骨架，視覺等設計稿**。

### 怎麼測試

- LIFF 頁面：手機 LINE 內打開看到自己的 LINE UID
- 訂閱 API：用 Postman POST → DB 看到新 row
- 取消 API：用 Postman DELETE → DB 看到 is_active=false

## Stage 4：TEP API 串接（拉廟宇資料）⏸ 待開始

### 在做什麼

把訂閱頁的廟宇列表**從假資料換成 TEP 提供的真實資料**。

### 為什麼需要這個 Stage

- 訂閱頁顯示的廟宇必須是真實的（不能寫死）
- 學會「呼叫 TEP API」這個技能，後面所有 TEP 整合都類似（報名、取消、活動詳情）
- 處理錯誤：TEP 掛掉怎麼辦、回應慢怎麼辦

### 任務內容

- 寫 `services/tep_client.py` 封裝 TEP API 呼叫
- 實作 `tep_client.list_temples()` 拉廟宇清單
- 訂閱頁從 TEP 拉資料（替換假資料）
- 錯誤處理（timeout、try/except）
- 簡單快取（避免每次都打 TEP）

### 關鍵概念：API Client 模式

所有 TEP 呼叫都要透過 `tep_client.py`，**不要散落在各個 blueprint**。這樣未來：

- 換 TEP 的 base URL 只改一處
- 加認證 header 只改一處
- 加 retry 邏輯只改一處

### 怎麼測試

- 訂閱頁顯示的廟宇來自 TEP（不是寫死）
- 故意關掉 TEP → 顯示友善錯誤訊息
- 多次重新整理 → 不會每次都打 TEP（有快取）

## Stage 5：活動推播（Flex Message）⏸ 待開始

### 在做什麼

寫一個 API，**廟方在 TEP 觸發後，本系統會發 Flex Message 給訂閱者**。

### 為什麼需要這個 Stage

- 推播是廟方→信眾的核心通訊管道
- Flex Message 比純文字訊息更有設計感、可互動（按鈕）
- 整合「訂閱關係」+「LINE Push API」+「TEP 觸發」三個系統

### 任務內容

- 寫 `services/flex_builder.py` 組活動邀請卡 JSON
- 寫 `services/push_service.py` 推播服務
- 寫 `POST /internal/push/event` API（給 TEP 呼叫）
- X-API-Key 驗證（不能讓任何人都能觸發推播）
- 查訂閱者 → 批次推播邏輯
- 推播結果寫進 push_logs
- 失敗重試（最多 3 次）

### 關鍵概念：multicast vs push

- **push**：發給單一使用者
- **multicast**：一次發給多個使用者（最多 500/批），效率高

推播訂閱者要用 multicast，不能 for 迴圈打 push（會超過 rate limit）。

### 怎麼測試

- curl 模擬 TEP 觸發 → 自己 LINE 收到 Flex 卡片
- push_logs 看到完整紀錄
- 故意把 LINE token 改錯 → 重試 3 次後失敗、紀錄 error

## Stage 6：報名互動（postback）⏸ 待開始

### 在做什麼

使用者按 Flex 上的「我要參加」→ **真的報名到 TEP**。

### 為什麼需要這個 Stage

- 完成「廟方→信眾→廟方」的雙向流（資料回到 TEP）
- 學 postback 互動（LINE 按鈕點擊事件）
- 處理「使用者沒填過個資」的流程：先導去補資料、再回來報名

### 任務內容

- 處理 postback event
- 解析 postback data（`action=register&event_id=X&response=yes`）
- 檢查使用者有沒有填過 phone
    - 沒填 → 推 LINE 訊息引導去補資料 LIFF
    - 已填 → 直接呼叫 TEP register API
- 寫 `tep_client.register_event()`
- 寫 action_logs 紀錄
- 推送確認訊息（成功 / 失敗）
- 寫補資料 LIFF（姓名 + 電話）
- 失敗處理（TEP 掛掉、名額滿、活動截止）

### 為什麼需要補資料 LIFF

TEP 報名 API 要 `name` + `phone`，但 LINE 只給 `displayName` + `userId`。所以**首次報名前要彈 LIFF 收集姓名電話**，之後從 line_users 取用。

### 怎麼測試

- 收到推播後按「我要參加」→ TEP 看到報名
- 沒填過 profile 的人 → 被導去填表 → 填完繼續報名
- 故意關 TEP → 收到「系統忙碌請稍後」訊息
- action_logs 有完整紀錄

# 七、資料庫 Schema

## 7.1 四張表

```
line_users
├── id, line_uid (UK), display_name, picture_url, phone (nullable)
├── follow_status (active / unfollowed)
└── followed_at, last_seen_at, created_at, updated_at

subscriptions
├── id, line_user_id (FK), tep_temple_id
├── is_active, subscribed_at
└── 複合唯一索引：(line_user_id, tep_temple_id)

action_logs
├── id, line_user_id (FK), action_type
├── request_payload (JSON), tep_response (JSON)
├── status (pending / success / failed)
└── created_at

push_logs
├── id, line_user_id (FK), tep_event_id
├── flex_payload (JSON), status (queued / sent / failed)
└── sent_at, error_message
```

## 7.2 關鍵設計決策

- `tep_temple_id`、`tep_event_id` 不是真正的外鍵，只是指向 TEP MySQL 裡的 ID
- 所有「使用者」概念用 `line_uid`（不是 LINE 帳號 email、不是 TEP 的 user_id）
- 軟刪除：`subscriptions.is_active=false` 而不是真的 delete
- JSON 欄位：用 PostgreSQL 的 JSONB 型別，可以索引和查詢

## 7.3 不會建的表

- ❌ `temples`（在 TEP）
- ❌ `events`（在 TEP）
- ❌ `registrations`（在 TEP）
- ❌ `admins`（在 TEP）
