前一篇「LINE Login OAuth 除錯馬拉松」 把 LINE 登入端到端打通。這篇是收尾三件事： ① 鏡像 LINE 結構實作 Google Login，② 前端「我的」頁加上兩顆品牌登入按鈕， ③ 清診斷 log、把整包 LINE + Google OAuth commit + push。

做了什麼

1Google Login OAuth（鏡像 line_login.py）
新增 google_login.py，4 個函式對照 LINE 版： get_authorize_url（導 accounts.google.com/o/oauth2/v2/auth，scope openid email profile）、 exchange_code_for_tokens（server-to-server 換 token）、 verify_id_token（打 Google tokeninfo 並檢查 audience = 我們的 client_id）、 generate_state（CSRF 防護）。 routes.py 加 3 個 endpoint：authorize / callback / bind，與 LINE 共用同一套 state cookie + JWT 機制。
2前端兩顆品牌登入按鈕
「我的」頁 LoginView 加上「使用 LINE 登入」「使用 Google 登入」兩顆按鈕， 統一白底 + 官方品牌 SVG logo（LINE 綠泡泡 / Google 四色 G）。 按鈕 href 用絕對網址指向後端 authorize —— OAuth 的 authorize + callback 必須同源才能對得上 state cookie，不能走 /api 代理。
3清診斷 log + commit + push
移除 LINE 除錯時埋的診斷訊息：前端 MemberProvider.tsx 5 行 console.log('[MemberProvider]…')、 後端 routes.py 2 段 logger.info('[oauth/…]…')（保留正式的 logger.error）。 刻意只 git add 10 個 OAuth 相關檔（不用 -A，避免把筆記 txt / 系統架構書 html 掃進來）， 確認無 .env 後 commit e73c1c0 → push origin/feat/line-auth。

設計重點

Why · 為什麼登入按鈕要用絕對網址
其他 API 都走相對 /api（Next 同源代理，避 CORS）。但 OAuth 例外： authorize 種的 oauth_state cookie 必須由 callback 同一個 origin 讀回來驗證。 若按鈕走代理，cookie 的 domain 對不上 → 永遠 invalid_state。所以這兩顆按鈕直接指後端 origin。
Security · audience 檢查不可省
verify_id_token 一定要比對 aud == GOOGLE_CLIENT_ID。 少了這步，別的 app 簽發的合法 Google token 也能拿來登入我們的站（token 替換攻擊）。
