    開始前 · 只要確認一件事

    登入流程裡最容易壞、又最不明顯的，就是 port 沒對齊。先花 10 秒確認：

    項目值要相等
    前端 npm run dev 印出的網址http://localhost:3000（看你實際的）兩者 port 必須一樣
    backend/.env 的 WEBAPP_URLhttp://localhost:3000
    為什麼重要： 登入成功後，後端會把你導回 WEBAPP_URL。如果它寫 3001、你前端開在 3000，就會「登入明明成功，卻跳到一個打不開的頁面」。對不上的話，改 .env 的 WEBAPP_URL 後重啟後端再測。

    實際登入 · 三個動作

    動作 1 · 開無痕視窗進站

    按 Ctrl+Shift+N 開無痕 → 網址列輸入你前端的位址（如 http://localhost:3000）→ 點進「我的」頁。

    務必無痕。 一般視窗的瀏覽器擴充（趨勢科技 URL 掃描）會搶先讀 callback、吃掉 OAuth 的 state cookie，害你卡 invalid_state。這是你 LINE 登入時踩過的同一個雷。

    動作 2 · 按「使用 Google 登入」

    頁面會跳轉到 Google 的授權畫面（網址變成 accounts.google.com/...）。

    動作 3 · 選帳號 → 允許

    選你那個有加進 Test users 的 Gmail → 按「繼續 / 允許」。然後盯著網址列看它導回哪裡（下一章就是判讀那串網址）。

    判讀 · 網址列會告訴你一切

    Google 把你導回前端時，網址後面會掛一段參數。那一段就是成績單。對照下表：

    導回的網址長這樣意思下一步
    ?login_success=true&token=eyJ... 登入成功，後端發了 JWT 往下做驗證
    ?login_error=invalid_state state cookie 被擋（多半是擴充） 換無痕重來
    ?login_error=server_error 後端換 token 時出錯（常是容器連不到 Google） 看後端 log / 重啟 Docker
    ?login_action=merge&provider=line 這個 email 已經用 LINE 註冊過 → 觸發帳號合併 C 方案 正常分支，見下方說明
    停在 Google 頁，紅字 redirect_uri_mismatch Console 的 redirect URI 跟 .env 不一致 對齊兩邊網址
    表 1 · 看到哪一行，就照「下一步」那欄做。
    關於 merge 那行： 如果你之前已經用「同一個 email 的 LINE」登入過，現在改用 Google，系統不會亂合併，而是導到合併頁問你要不要綁定 —— 這是正確行為，不是錯誤。想測「純新 Google 帳號」的話，用一個沒登入過的 Gmail。

    確認 · 三個成功訊號

    看到 login_success=true 還不算完，要這三個都對才是端到端通過：

    訊號 1 · 網址
    login_success
    網址列出現 =true&token=...
    訊號 2 · 畫面
    顯示名字
    「我的」頁出現 Google 名字 + 頭像
    訊號 3 · 持久
    重整還在
    按 F5 後仍是登入狀態

    用 DevTools 再確認一次（可選但推薦）

    按 F12 開 DevTools → Application 分頁 → 左邊 Local Storage → 點你的網域，找這個 key：

    lyz_member_token   // 有值 = token 真的存下來了

    有這個 key、值是一長串 eyJ...，就代表前端確實接到並保存了 token，重整後才能維持登入。

    三個訊號都對 = 通過
