任務目標

在學長的 templeuser（緣引拜拜通 Web App）實作 LINE Login OAuth， 讓信眾能用 LINE 登入、前端正確顯示登入狀態。

OAuth 後端流程很快就寫完（authorize / callback / bind），但「前端看不到登入狀態」這件事 像剝洋蔥 —— 一個功能底下藏了 8 層獨立問題，環境、瀏覽器、Docker、Next.js 設定各踩一個。 這份 log 把每一層記下來，給未來的自己（跟學長）看。

8 層問題（依發現順序）

1實作 · OAuth 後端 + DB 欄位
新增 line_login.py（authorize URL / 換 token / 驗 id_token）+ routes 三個 endpoint。 members 表加 email / google_sub / push_enabled、line_user_id 改可空（為多 OAuth + 帳號合併鋪路）。 修了文件原稿一個 bug：callback 用了 current_app 卻沒 import → 補上。
2404 · next.config rewrites 缺 members
前端打 /api/members/me 回 404。原因：next.config.mjs 的 rewrites 漏了 members 跟 needs（只有原本 6 模組 + analytics/admin/auth）。 → 補上兩條 rewrite。整個 members 功能其實一直沒通過代理測過。
3前端沒接 callback token
LINE callback 把 token 放 URL 導回，但 MemberProvider 只讀 localStorage、不讀 URL。 → 在 bootstrap 最前面加「讀 URL token → 存 localStorage → replaceState 清網址」。
4Hydration Error · 巢狀 <head>
(consumer)/layout.tsx 在巢狀 layout 渲染原生 <head>（App Router 不允許， 會變成 <body> 子節點）→ hydration 失敗。 → 拿掉 <head> 包裝，<link> 直接渲染（Next 自動提升到 head）。
5invalid_state · 趨勢科技擴充套件
callback 回 ?login_error=invalid_state（CSRF state cookie 對不上）。 元兇是瀏覽器的趨勢科技 URL 掃描器擴充（TISProUrlManualScannerLib）搶先掃 callback、吃掉 cookie。 → 用無痕視窗（停用擴充）繞過。
6server_error · Docker 容器連不到外網
後端 log：Failed to resolve 'api.line.me' (Temporary failure in name resolution)。 容器換 token 時連不到 LINE API —— Docker 容器 DNS 壞了（跟先前 node:20 build timeout 同根因）。 → 完整重啟 Docker Desktop（WSL2 DNS 復原）。重啟後出現 login_success=true&token=...，OAuth 真的通了。
7CORS · NEXT_PUBLIC_API_URL 絕對網址
.env.local 把 NEXT_PUBLIC_API_URL 設成 http://localhost:5001/api（絕對）→ 瀏覽器跨域打後端 → 後端 CORS 只允許 3001 不允許 3000 → 擋掉，連 me() 驗證也掛 → token 被清。 → 改成相對 /api，走 Next 同源代理。
8401 · 代理跑去 prod 後端（最終根因）
改 /api 後變 401。實測：同一 token 打本地後端 200、打代理 401 → 代理把請求轉去了別的後端。 根因：next.config.mjs 的 BACKEND_INTERNAL_URL || 'https://yuanyinb.zeabur.app'， 本地 dev 沒設這變數 → 代理 fallback 到 prod Zeabur 後端。 本地簽的 token 被 prod 用不同 JWT secret 驗 → 401。 → .env.local 加 BACKEND_INTERNAL_URL=http://localhost:5001（待重啟驗證）。

關鍵心得

Lesson · 「前端看不到登入」可能離前端很遠
症狀在前端（看不到名字），但 8 層根因散落在 next.config、瀏覽器擴充、Docker DNS、env 設定。 靠「同一 token 分別打本地 vs 代理」這種對照測試，才把最後的代理指向問題逼出來。 診斷 log（前端 console + 後端 logger）是這次定位的關鍵 —— 沒有它只能瞎猜。
Caveat · dev 代理預設指向 prod 很危險
BACKEND_INTERNAL_URL || 'https://yuanyinb.zeabur.app' 這種 fallback， 會讓沒設變數的本地 dev 靜默打到 prod —— demo 帳號建在 prod DB、token secret 不符。 值得跟學長提：本地 .env.local 應該明確設 BACKEND_INTERNAL_URL。
