啟動 Docker Desktop
為什麼要做
Docker 是一個「虛擬環境管理工具」。你的 MySQL 資料庫和 Flask 後端都跑在 Docker 裡面。Docker Desktop 沒啟動的話，這兩個服務就不存在。

用生活比喻：Docker Desktop 像「啟動發電機」。不開發電機，餐廳（後端）和冰箱（資料庫）都不會動。
操作
方法 A：按 Windows 鍵 → 搜尋「Docker Desktop」→ 點開
方法 B：PowerShell 執行： 
Start-Process "C:\Program Files\Docker\Docker\Docker Desktop.exe"
等右下角系統匣的鯨魚圖示不再轉圈（變靜止）= 啟動完成。通常 30 秒到 2 分鐘。
預期結果
PowerShell 跑 docker info 印出一大堆資訊（不是 error）→ Docker 啟動成功。

如果出現「Docker Desktop is unable to start」→ Docker Desktop 還沒完全啟動，再等一下。

啟動 MySQL + Flask 後端（Docker 裡的兩個服務）
為什麼要做
MySQL 是資料庫，存帳號、廟宇、神明等所有資料。
Flask 後端 是 API 伺服器，處理登入邏輯、查詢資料、驗證身分。

它們跑在 Docker 容器裡（像兩個獨立的小電腦）。你不需要自己安裝 MySQL，Docker 幫你處理。

用生活比喻：MySQL = 冰箱（存食材），Flask = 廚房（做菜）。兩個都要開，餐廳才能營業。

預期結果
✔ Container yuanyin-mysql    Healthy
✔ Container yuanyin-backend  Started
Healthy 代表 MySQL 完全啟動、可以連線了。

確認： 
docker-compose ps
應該看到 yuanyin-mysql 和 yuanyin-backend 都是 Up。

確認環境變數 .env 設好了
為什麼要做
程式碼裡不會寫死密碼和 API Key（寫死的話推上 GitHub 就洩漏了）。這些敏感資訊放在 .env 檔案裡——每台電腦各自設、不推上 git。

用生活比喻：.env 就像餐廳的「保險箱密碼」。每間分店各自設、不會寫在菜單上。

有兩個 .env：
· C:\templeuser\.env → 給 Docker 用（後端 + MySQL）
· C:\templeuser\frontend\.env.local → 給前端用
確認指令
# 後端 .env（確認 LINE Login 變數有設）
Select-String "LINE_LOGIN" C:\templeuser\.env

# 前端 .env.local（確認代理指向本地後端）
Select-String "BACKEND_INTERNAL_URL" C:\templeuser\frontend\.env.local
預期結果
後端 .env 應該看到：

LINE_LOGIN_CHANNEL_SECRET=（一串英數字）
LINE_LOGIN_CALLBACK_URL=http://localhost:5001/api/members/oauth/line/callback

BACKEND_INTERNAL_URL=http://localhost:5001

確認資料庫表存在（首次啟動才需要）
為什麼要做
MySQL 第一次啟動時是空的——沒有 members、temples、deities 任何一張表。需要手動跑 schema.sql 把所有表建起來。

什麼時候要做？
· 第一次跑 Docker
· 刪除過 MySQL volume（docker volume rm）
· 換了新電腦

什麼時候不用做？
· 之前跑過、表已經存在 → 跳過這步
指令
先檢查表有沒有： 
docker-compose exec yuanyin-mysql mysql -u root -p"yuanyin-root" yuanyin -e "SHOW TABLES;"
如果有印出 members、temples 等 → 表已存在，跳過。

如果是空的或出現 Unknown database → 跑 schema.sql： 
Get-Content backend/schema.sql | docker-compose exec -T yuanyin-mysql mysql -u root -p"yuanyin-root"
注意：這裡不指定資料庫名稱（因為 schema.sql 裡面有 CREATE DATABASE 和 USE yuanyin）。
預期結果
跑完後再確認： 
docker-compose exec yuanyin-mysql mysql -u root -p"yuanyin-root" yuanyin -e "SHOW TABLES;"
應該看到約 10 張表：analytics_events、deities、favorites、guides、members、needs、schema_migrations、temples、temple_deities、wishes、worship_logs 等。

確認後端活著
為什麼要做

確認前端代理通到後端
為什麼要做
前端（Next.js, port 3000）跟後端（Flask, port 5001）是兩個不同的程式。瀏覽器不能直接打 5001（會有 CORS 問題），所以 Next.js 設了代理：/api/* 的請求會被自動轉到 Flask。這一步確認代理有在運作。
指令
curl.exe -o NUL -w "proxy %{http_code}`n" http://localhost:3000/api/members/me
預期結果
proxy 401 → 代理成功把請求轉到後端（401 = 後端收到了、只是沒 token）

如果是 404 → 代理沒設好（next.config.mjs 的 rewrites 有問題）
如果是 500 → 代理轉到了錯的後端

清除舊登入狀態（確保乾淨測試）
為什麼要做
瀏覽器的 localStorage 可能殘留舊 token。舊 token 如果過期或無效，會干擾測試——前端以為你「已登入」，結果打 API 失敗、又把你踢回「未登入」，造成混亂。
操作
方法 A：用無痕視窗（Ctrl+Shift+N）
方法 B：F12 → Application → Local Storage → localhost:3000 → 刪除 lyz_member_token
預期結果
Local Storage 裡沒有 lyz_member_token
5
啟動 LINE Login OAuth
為什麼要做
這是整個流程的起點。你打後端的 /api/members/oauth/line/authorize，後端會組一個 LINE 授權 URL、把你導過去。

為什麼不是打前端的 /api/？
因為 OAuth 的 authorize 步驟需要「server-side redirect」（伺服器直接導向），不是 AJAX。前端代理不適合處理 redirect，所以直接打後端。
操作
瀏覽器輸入：http://localhost:5001/api/members/oauth/line/authorize
預期結果
瀏覽器跳到 LINE 的綠色授權頁面。
網址列變成 https://access.line.me/oauth2/v2.1/authorize?...

如果看到錯誤頁面 → Channel ID 或 Callback URL 設錯了
6
在 LINE 授權
為什麼要做
這是 OAuth 的核心——使用者「親自同意」把自己的資料（名字、頭像、email）給你的 App。LINE 不會在使用者不知情的情況下洩漏資料。

OAuth 的安全原則：你的 App 永遠拿不到使用者的 LINE 密碼。你只拿到一個「臨時授權碼」（code），用這個 code 去跟 LINE 換資料。
操作
在 LINE 授權頁面，用手機掃 QR code 或輸入帳密，然後按「允許」
預期結果
LINE 把你導回後端的 callback URL：
http://localhost:5001/api/members/oauth/line/callback?code=xxx&state=xxx

你不會看到這個頁面——後端收到後會立刻再導回前端。整個過程一秒內完成。
7
觀察導回前端的結果
為什麼要做
後端收到 LINE 的 code 後，做了以下事情（你看不到、但在背後發生）：
1. 用 code 跟 LINE 換 access_token + id_token
2. 驗證 id_token（確認是 LINE 簽的、沒被篡改）
3. 從 id_token 取出你的 LINE UID、名字、頭像
4. 在 members 表建立或更新你的帳號
5. 簽發一個 JWT token（這是「你是誰」的證明）
6. 把你導回前端首頁，token 放在 URL 裡
預期結果
瀏覽器網址列短暫出現：
http://localhost:3000/?login_success=true&token=eyJ...

然後自動變乾淨成 http://localhost:3000/
（MemberProvider 把 token 存到 localStorage、然後清掉 URL）

操作
點底部導覽列的「我的」

常見問題排除

問題：Docker Desktop is unable to start
Docker Desktop 沒有完全啟動。等鯨魚圖示不轉再跑指令。
如果一直起不來：可能缺 WSL2 → 跑 wsl --install → 重開機。
問題：Table 'yuanyin.members' doesn't exist
資料庫是空的、schema.sql 沒跑過。回 Step 0e 跑 schema.sql。
問題：Step 2 代理回 404 而不是 401
Next.js 的 rewrites 沒有轉發 /api/members/*。確認 next.config.mjs 的 rewrites 有包含 members。
問題：Step 2 代理回 401（但直打後端帶 token 回 200）
代理指向了 Zeabur 正式後端、不是本地後端。
確認 frontend/.env.local 有 BACKEND_INTERNAL_URL=http://localhost:5001。
改完要 Ctrl+C → npm run dev 重啟（這個環境變數不支援熱重載）。
問題：LINE 授權頁出現 error（Channel ID 或 redirect URI 錯誤）
去 LINE Developers Console 確認：
· LINE Login Channel ID 跟 .env 裡的一致
· Callback URL 有加 http://localhost:5001/api/members/oauth/line/callback
問題：callback 回來變 login_error=server_error
後端跟 LINE 溝通失敗。跑 docker-compose logs yuanyin-backend --tail 30 看 error。
常見原因：Docker 容器 DNS 壞了（連不到 api.line.me）。
修法：docker-compose restart yuanyin-backend，或重啟 Docker Desktop。
問題：callback 回來變 login_error=invalid_state
CSRF state cookie 對不上。可能原因：
· 瀏覽器擴充套件（趨勢科技、防毒軟體）搶先掃 callback URL → 用無痕視窗重試
· 超過 10 分鐘才按「允許」→ cookie 過期 → 重走一遍
問題：登入成功但「我的」頁面還是 demo 表單
F12 → Application → Local Storage → 看有沒有 lyz_member_token。
· 沒有 → MemberProvider 的改動沒生效（確認 MemberProvider.tsx 有讀 URL token 的邏輯）
· 有 → 打 F12 → Network → 搜尋 me → 看 /api/members/me 狀態碼
  · 200 → 前端邏輯問題（罕見）
  · 401 → 回上面「代理回 401」那條

Part 2 · 系統解說（你到底做了什麼）

用「餐廳」比喻整個系統
想像你開了一間餐廳。

Web App（Next.js） = 餐廳的門面和裝潢。客人走進來看到菜單、看到擺設。
Web App 後端（Flask） = 廚房。客人看不到，但沒有廚房就沒有菜。
MySQL 資料庫 = 冰箱和倉庫。所有食材（資料）存在這裡。
LINE Gateway = 外送平台的接口。告訴外面的人「我們有新菜」。
TEP = 食材供應商（廟方資料）。

你今天做的「LINE Login」 = 幫餐廳裝了一台「會員卡機」。客人走進來刷 LINE，餐廳就知道你是誰、記住你喜歡什麼。

OAuth 流程全圖

信眾（你） │ │ ① 點「用 LINE 登入」 ▼ Flask 後端 /api/members/oauth/line/authorize │ │ ② 組 LINE 授權 URL → redirect ▼ LINE 伺服器 access.line.me/oauth2/v2.1/authorize │ │ ③ 信眾按「允許」→ LINE 給一個臨時授權碼（code） ▼ Flask 後端 /api/members/oauth/line/callback?code=xxx │ │ ④ 用 code 跟 LINE 換 id_token（server-to-server，信眾看不到） │ ⑤ 驗 id_token → 拿到 LINE UID + 名字 + 頭像 │ ⑥ 在 MySQL 建立 member 帳號 │ ⑦ 簽發 JWT token │ ⑧ redirect 到前端，帶 token ▼ Next.js 前端 localhost:3000/?token=eyJ... │ │ ⑨ MemberProvider 讀 URL token → 存 localStorage │ ⑩ 打 /api/members/me → 經代理到後端 → 驗 JWT → 回傳資料 │ ⑪ 顯示「邱識瑋 · LINE 會員」 ▼ 信眾看到自己的名字 🎉

你改了什麼、為什麼改

JWT Token 到底是什麼

你在 URL 裡看到的那串 eyJhbGciOiJIUzI1NiIs... 就是 JWT（JSON Web Token）。它是一張「數位身分證」。

JWT 分三段，用 . 分隔：

eyJhbGciOiJIUzI1NiIs... ← Header（演算法資訊） eyJmcmVzaCI6ZmFsc2Us... ← Payload（你的資料） brB_BsLBlMZMU2g_cqyd... ← Signature（防偽造簽章） Payload 解開後長這樣： { "kind": "member", ← 這是會員 token（不是 admin token） "memberId": 1, ← 你在 DB 裡的 ID "lineUserId": "U8a2e..", ← 你的 LINE UID "exp": 1780519332 ← 過期時間（8 小時後） } 重點： · JWT 不是加密的（任何人都能解開看內容） · 但只有後端能「簽」（因為只有後端知道 secret） · 所以它能證明「這張身分證是後端發的、沒被竄改」 · 就像你的身分證不是加密的（大家都看得到名字） 但只有政府能發（有防偽鋼印）
